# CPA Launcher 拖拽导入 CPA 认证目录设计

## 背景

当前用户需要通过 CPA 网页端上传本地 token 凭证文件，流程偏慢，也不符合桌面启动器的使用习惯。对于本地使用场景，启动器可以提供更顺手的入口：将本地 `.json` 凭证文件直接拖到启动器窗口，由启动器放入当前 CPA 配置对应的认证目录。

该能力仅服务于本地桌面流程，不替代 CPA 的网页管理能力，也不改变 CPA 对认证目录的解释方式。启动器只负责把文件投递到当前配置已经指定的 `auth-dir`。

## 目标

- 支持将一个或多个本地 `.json` 文件拖入主窗口任意位置。
- 启动器根据当前 `config.yaml` 解析得到 `auth-dir`，并将文件导入该目录。
- 同名文件直接覆盖，不弹确认框。
- 使用明显但不打断的整窗拖拽交互和轻提示反馈。
- 在第一次启动 CPA 之前也可以完成导入；若目标目录不存在，则在导入时按需创建。

## 非目标

- 不支持通过拖拽导入非 `.json` 文件。
- 不支持拖入文件夹。
- 不支持拖入时手动选择目标路径。
- 不支持部分接收混合文件列表；如果拖入内容中包含非 `.json` 文件，则整次拒绝。
- 不在本次设计中扩展为网页端同步、远程导入或凭证内容校验。

## 体验原则

- 以实际体验优先：用户只需要“拖进窗口，松手完成”。
- 以配置一致性为底线：启动器和 CPA 都只认当前 `config.yaml` 中的同一个 `auth-dir`。
- 以反馈清晰但不打断为标准：避免阻塞式弹窗，优先使用整窗状态提示和自动消失的轻提示。

## 方案概览

采用“平衡型”方案：

- 主窗口整体支持拖放，而不是单独设置局部投递区。
- 用户拖入窗口时显示整窗半透明蒙版和居中接收卡片。
- 合法拖拽显示可接收态；非法拖拽显示警示态。
- 放下后立即导入到当前配置推导出的 `auth-dir`。
- 导入结果通过轻提示和诊断区日志反馈。

## 交互设计

### 空闲态

- 主窗口保持现有布局。
- 在“认证目录”附近增加一条较轻的辅助说明，例如“支持将 `.json` 凭证拖入窗口直接导入”。

### 可接收态

- 用户将一个或多个 `.json` 文件拖入主窗口任意位置时，窗口上方出现整窗蒙版。
- 蒙版中央展示浮起卡片，主文案为“松手导入到当前 CPA 认证目录”。
- 次文案展示当前解析出的 `auth-dir` 路径。
- 视觉上使用轻微阴影、柔和高亮和适度缩放，表现为桌面应用的接收态，而非网页上传框。

### 不可接收态

- 当拖入内容不是全 `.json` 文件、包含文件夹、未解析到有效 `auth-dir`，或当前配置无效时，仍显示蒙版，但切换为警示样式。
- 警示态主文案按原因展示，例如“仅支持导入 .json 凭证文件”或“当前未配置可用的认证目录”。

### 完成反馈态

- 导入完成后蒙版立即消失。
- 窗口右上角或顶部显示一个自动消失的轻提示，持续约 2 到 3 秒。
- 成功时提示示例：“已导入 3 个凭证文件”。
- 部分失败时提示示例：“已导入 2 个，失败 1 个”。
- “认证目录”这一行可增加一次轻微高亮闪动，强化“已经放到这里”的反馈。

## 核心行为规则

### 文件接收规则

- 仅接受本地文件拖拽。
- 仅接受扩展名为 `.json` 的文件，大小写不敏感。
- 允许一次拖入多个 `.json` 文件。
- 如果拖入列表中包含任何非 `.json` 文件或文件夹，则整次拖拽视为不可接收，不做部分导入。

### 导入规则

- 目标目录始终为当前 `config.yaml` 解析出的 `auth-dir`。
- 不允许为导入动作设置单独的默认路径或备用路径。
- 同名文件直接覆盖，不弹确认框。
- 导入时如果目标目录不存在，则由启动器按 `auth-dir` 路径即时创建。
- 导入动作不依赖 CPA 当前是否已经启动。

### 配置一致性规则

- 启动器不自行定义认证目录。
- 启动器和 CPA 对认证目录的理解都来自同一个 `config.yaml`。
- 如果用户修改 `config.yaml` 中的 `auth-dir`，启动器后续拖拽导入应自动跟随新的解析结果。
- 如果当前配置无法解析出有效 `auth-dir`，则禁止导入，不进行路径猜测或隐式回退。

## 架构设计

### 视图层

`MainWindow.xaml` 与 `MainWindow.xaml.cs` 负责窗口级拖拽事件和拖拽视觉层：

- 为主窗口开启 `AllowDrop`。
- 处理 `DragEnter`、`DragOver`、`DragLeave`、`Drop`。
- 在 XAML 中新增覆盖整窗的拖拽蒙版层与轻提示层。
- 将拖拽状态、可接收状态和目标路径等数据绑定到 ViewModel。

### ViewModel 层

`MainViewModel` 负责拖拽状态和导入反馈编排：

- 维护是否显示拖拽蒙版、当前是否可接收、拖拽提示文案、提示类型、轻提示文本等状态。
- 调用导入服务执行文件复制。
- 将结果写入诊断日志。
- 在导入成功或部分失败后触发轻提示展示。

### 服务层

新增 `TokenImportService`，职责保持单一：

- 校验待导入文件列表是否合法。
- 解析和归一化目标路径输入。
- 按需创建目标目录。
- 复制文件并记录覆盖、成功、失败结果。
- 返回包含成功数量、失败数量、覆盖文件列表和错误详情的结果对象。

## 数据流

1. 用户将文件拖入主窗口。
2. 视图层读取拖拽数据，判断是否为文件列表。
3. ViewModel 根据当前 `AuthDirectory` 和文件列表计算拖拽状态。
4. 用户松手后，ViewModel 调用 `TokenImportService` 执行导入。
5. 服务层创建目标目录（如有需要），逐个覆盖复制文件并汇总结果。
6. ViewModel 更新轻提示、诊断日志和必要的局部高亮状态。

## 错误处理

### 拖拽前置错误

- 当前未配置有效 `config.yaml`：不可接收。
- 当前无法解析 `auth-dir`：不可接收。
- 拖入内容中存在非 `.json` 文件或文件夹：不可接收。

### 导入过程错误

- 目标目录不存在：自动创建后继续导入。
- 单个文件复制失败：不中断其余文件，最终汇总为部分失败。
- 所有文件都失败：显示失败轻提示，并在诊断区输出详细原因。

### 反馈要求

- 非法拖拽不弹阻塞式对话框。
- 导入错误优先通过轻提示摘要说明，通过诊断区记录细节。
- 对于被覆盖文件，诊断区记录“已覆盖同名凭证：文件名.json”。

## 测试设计

### 单元测试

为 `TokenImportService` 添加测试，覆盖至少以下场景：

- 导入多个合法 `.json` 文件成功。
- 目标目录不存在时自动创建并成功导入。
- 同名文件被正确覆盖。
- 混合扩展名文件列表被整体拒绝。
- 文件夹输入被拒绝。
- 部分文件复制失败时返回正确统计和错误详情。

### 手工验证

- 未配置 `config.yaml` 时拖拽行为。
- 已配置但 `auth-dir` 目录尚不存在时的首次导入。
- 主窗口任意位置拖入多个 `.json` 文件。
- 混合拖入 `.json` 与其他文件时的警示态。
- 已有同名文件时的静默覆盖和日志记录。
- 轻提示是否自动消失且不抢焦点。
- 配置变更后 `auth-dir` 路径展示与导入目标是否同步更新。

## 实施建议

优先顺序如下：

1. 先抽出 `TokenImportService` 和结果模型，并补单元测试。
2. 再在 `MainViewModel` 中接入导入状态与提示状态。
3. 最后在 `MainWindow` 中补整窗拖拽事件、蒙版层和轻提示层。

这样可以先把文件系统逻辑和配置一致性做稳，再处理 UI 表现。

## 风险与取舍

- 整窗拖拽层会引入额外 UI 状态，但能显著提升可发现性和完成感，值得增加这部分复杂度。
- 由于首版不做凭证内容校验，启动器无法判断 `.json` 内部是否真的是 CPA 可用 token；这部分仍由 CPA 运行时承担最终解释。
- 如果未来 CPA 对认证文件命名或结构有更严格约束，可在服务层追加规则，而不需要重做整套交互。

## 最终结论

该功能应实现为：主窗口整窗支持拖入一个或多个 `.json` 文件，启动器始终依据当前 `config.yaml` 解析得到的 `auth-dir` 执行导入；当目录不存在时按需创建；同名静默覆盖；通过整窗蒙版和轻提示提供清晰、好看且不打断的反馈。
